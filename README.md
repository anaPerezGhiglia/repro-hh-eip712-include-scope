# Repro: Hardhat 3 EIP-712 collector throws `HHE818` for a same-named struct in a non-included source

Minimal dual-toolchain reproduction for an EIP-712 type-collector scoping bug in `hardhat@3.6.0`. The same project, with equivalent config, runs cleanly under Forge — proving the divergence is a Hardhat-side scoping decision rather than a project-level ambiguity.

## TL;DR

`test.solidity.eip712Types.include` (Hardhat) and `[bind_json].include` (Forge) are both declarations of which Solidity files supply EIP-712 type definitions. The expectation — and Forge's actual behavior — is that `include` bounds the set of structs that can register names and trigger name-collision errors.

Hardhat does not honor this scope. `collectEip712CanonicalTypes` (`solidity-test/eip712/index.ts:91-116` in `3.6.0`) walks every source's AST and pushes its structs into the registry unconditionally; `include`/`exclude` is only consulted to populate `selectedNames` at emit time. A same-named struct in an *unincluded, unimported* file collides at `canonicalize.ts:indexByName` with the selected struct and throws `HHE818` — even though the user explicitly scoped that other file out.

Forge does not have this divergence — `[bind_json].include` controls the `target_files` set, and only structs in `target_files` enter the registry. See [`crates/forge/src/cmd/bind_json.rs`](https://github.com/foundry-rs/foundry/blob/master/crates/forge/src/cmd/bind_json.rs) (≈ lines 159-213, 280-286).

## Project shape

```
.
├── contracts/
│   ├── A.sol      ← in eip712Types.include
│   │              defines `library A { struct Order { address, uint256 } }`
│   └── B.sol      ← NOT included, NOT imported by A.sol or the test
│                    defines a same-named `struct Order { bytes32, bool }`
├── test/
│   └── Foo.t.sol  ← uses vm.eip712HashStruct(...) against A.Order
├── hardhat.config.ts     ← eip712Types: { include: ["contracts/A.sol"] }
├── foundry.toml          ← src/test/solc/evm_version only
└── remappings.txt        ← forge-std/=node_modules/forge-std/src/
```

`contracts/A.sol` has zero imports — `contracts/B.sol` is not reachable from the selected file by any Solidity import path.

## Reproducing

### Recommended: open in the bundled devcontainer

The repo ships a `.devcontainer/devcontainer.json` that pins a Debian 12 (bookworm) base image so the GLIBC is new enough to run the latest stable Foundry. `postCreateCommand` installs Foundry via `foundryup` and runs `npm install` for you.

In VS Code: "Reopen in Container" → wait for `postCreateCommand` to finish → then:

```bash
npm test               # Hardhat — fails with HHE818
forge test             # Foundry — passes
```

If you'd rather skip the devcontainer, the project needs:

- Foundry stable (`foundryup` against current libc — needs GLIBC `≥ 2.32`, so Ubuntu 22.04 / Debian 12 or newer)
- `npm install` to pull `hardhat@^3.6.0` + `forge-std`

Then:

```bash

```

## Self-test: confirm it's the scope issue

Delete `contracts/B.sol` and re-run `npm test` — Hardhat passes too. Add `B.sol` back, even though no source file imports it and no test references it, and Hardhat breaks again. Forge is unaffected either way.

## A note on the cheatcode form used in the test

`vm.eip712HashStruct(typeNameOrDefinition, abiEncodedData)` accepts either a bare type **name** (resolved at runtime through the file generated by `forge bind-json`) or a literal **type definition** string. The test in this repro uses the **definition** form:

```solidity
vm.eip712HashStruct("Order(address user,uint256 amount)", abi.encode(o));
```

That choice does two things at once. First, it keeps the repro independent of `forge bind-json`'s generated file — no extra step, no `JsonBindings.sol`, no `fs_permissions`. Second, and more importantly, it makes visible just **how early Hardhat fails**: the test never actually runs under Hardhat, so what shape its cheatcode call took is irrelevant. `HHE818` is thrown at collection time, before any test code executes. The bug is structural, not dependent on what the user is actually trying to hash.

### Optional: also validate the name-form path under Forge

In a real project, the name form is more common (it's what the bound `JsonBindings.sol` is for). To check that path too:

1. Add to `foundry.toml`:

   ```toml
   fs_permissions = [{ access = "read", path = "test/JsonBindings.sol" }]

   [bind_json]
   out = "test/JsonBindings.sol"
   include = ["**/A.sol"]
   ```

   Use the `**/A.sol` glob (or just `"A.sol"`). The literal `"contracts/A.sol"` form silently produces empty bindings — see the matching table below.

2. Add `contracts/=contracts/` to `remappings.txt` so Hardhat can also resolve the absolute import the generated file emits (`import {A} from "contracts/A.sol";`).

3. Generate the bindings:

   ```bash
   forge bind-json
   ```

   Inspect `test/JsonBindings.sol` — it contains exactly one `schema_Order = "Order(address user,uint256 amount)"` line and no entry for `B.sol`, even though `B.sol` declares a same-named struct in the same compilation set. That's the include-scope behavior Hardhat fails to reproduce, now visible as a file on disk.

4. Switch the cheatcode call in `Foo.t.sol` to the name form:

   ```solidity
   bytes32 h = vm.eip712HashStruct("Order", abi.encode(o));
   ```

5. `forge test` still passes (the cheatcode resolves the name through the bindings file); `npm test` still throws `HHE818`.
